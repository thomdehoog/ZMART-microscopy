# zenapi — ZEISS ZEN API microscope driver

`zenapi` drives a **ZEISS ZEN** microscope from an external Python process through the **ZEN API**
(a gRPC interface served by the ZEN API Gateway). It is a vendor sibling to the Leica
`navigator_expert` driver and deliberately mirrors its architecture — *connection + command
vocabulary + state readers*, with all tuning in profiles — while adapting the transport to ZEN's
fully-async, typed gRPC surface. The public API is **synchronous**, so operator notebooks keep the
thin 1–3-line invocation style used across the lab's drivers.

- **Author:** Thom de Hoog (ZMB, University of Zurich) · thom.dehoog@zmb.uzh.ch · thomdehoog@gmail.com
- **License:** MIT

  implemented and green. Server-facing details are transcribed from the

  **not yet validated against the `zen_api` wheel or a live gateway** — see [§10 Risks](#10-risks--bench-verify).

## 1. About ZEN API

ZEN API is ZEISS's interface for controlling and monitoring a running ZEN instance from an external
process. Key properties this driver relies on:

- **Transport:** gRPC over HTTP/2 with **mandatory TLS**, reached through a **ZEN API Gateway** — a
  separate application (installed via the ZEISS Microscopy Installer) that authenticates clients and
  fronts ZEN. The driver connects to the gateway host/port, not to ZEN directly.
- **Authentication:** a **global control token** passed as a gRPC metadata header
  (`control-token`) on every call. Generated by the gateway; stored at
  `%PROGRAMDATA%\Carl Zeiss\ZEN APIGateway\GlobalControlToken.txt`.
- **Async & streaming:** the official client is `asyncio`-native (`grpclib`). Unary RPCs are
  coroutines; monitoring RPCs (experiment status, pixel data) are **server-streaming async
  iterators**. `zenapi` hides all of this behind a blocking facade.
- **Service model:** a flat set of per-subsystem **service stubs** (no single "god" object). This
  driver uses the light-microscopy device services (`zen_api.lm.hardware.v2`: stage, focus,
  objective) and the acquisition services (`zen_api.acquisition.v1beta`: experiment control + status
  and pixel streams).
- **Units:** ZEN speaks **SI meters** on the wire. `zenapi` exposes **micrometers** publicly and
  converts at the boundary only.
- **Supervised vs unsupervised:** by default the ZEN UI is locked while the API controls the
  instrument. Enable *Unsupervised API Mode* (Tools ▸ Options ▸ ZEN API) for concurrent UI + API use.

Requires **ZEN ≥ 3.11** (the 2025/2026 API releases target ZEN 3.13/3.14).

## 2. Requirements & installation

On the ZEN acquisition PC:

1. **Install the ZEN API toolkit + gateway** via the ZEISS Microscopy Installer (ZMI). Enable the ZEN
   API in ZEN; enable *Unsupervised API Mode* if you want to keep using the UI while the API runs.
2. **Install the Python packages** into your environment:
   ```bash
   pip install <path>\zen_api-<version>-py3-none-any.whl   # ships with the toolkit; NOT on PyPI
   pip install grpclib
   ```
   (`zenapi` itself imports these lazily, so the driver and its offline tests import fine without them.)
3. **Generate a control token** from the gateway (system-tray icon, or
   `--generate-global-control-token`).
4. **Create `config.ini`** — copy `config.ini.example` and fill in the values (see [§3](#3-configuration)).
   **Never commit `config.ini`** (it holds the token); the driver's `.gitignore` already excludes it.

Python 3.10+ recommended (the driver uses `X | Y` type syntax).

## 3. Configuration

`connect()` resolves connection parameters from a `config.ini` and/or explicit keyword overrides.

`config.ini`:

```ini
[api]
host = 127.0.0.1
port = 5002
cert_file = C:\ProgramData\Carl Zeiss\ZEN APIGateway\Certificates\ZEN APIPersonalSigningRootCA.pem
control-token = <your global control token>
```

| Key | Meaning |
|---|---|
| `host`, `port` | ZEN API Gateway address (deployment-specific; the gateway default is often 5002). |
| `cert_file` | Gateway CA `.pem` used to verify the TLS connection. Relative paths resolve against the config file. |
| `control-token` | Global control token, sent as the `control-token` gRPC metadata header on every call. |

Connection **timing** defaults live in the `ZenApiProfile` (`config/profiles.py`, exported as
`ZEN_API`): `connect_timeout_s` (channel build) and `default_call_timeout_s` (per-RPC deadline).

## 4. Quick start

```python
import zenapi as drv

# 1. Connect (TLS + control-token handshake, then a ping to verify).
client = drv.connect("config.ini")

# 2. Configure the stage safety envelope (micrometers) once per session.
drv.apply_stage_limits_from_config(drv.load_stage_config("stage.json"))

# 3. Drive the hardware — synchronous, micrometers.
drv.move_xy(client, 1000, 2000)                      # µm
drv.move_z(client, 50)                               # µm
drv.set_objective(client, name="Plan-Apochromat 20x/0.8")

# 4. Read state.
print(drv.get_xy(client))                            # {'x_m':..,'y_m':..,'x_um':1000.0,'y_um':2000.0}
print(drv.get_objective(client))                     # {'index':1,'name':...,'magnification':20}

# 5. Acquire and persist.
exp = drv.load_experiment(client, "TileScan_10x")
acq = drv.acquire(client, exp)                       # blocks until the acquisition completes
saved = drv.save(client, acq, output_root, naming)   # copies the CZI into the output layout

# 6. (Optional) watch live progress.
for status in drv.monitor(client, exp):
    print(status["images_acquired_index"])

# 7. Close (stops the background event loop).
drv.close(client)
```

## 5. Core concepts

**`ZenClient` — the connection handle.** `connect()` returns a `ZenClient`; every command and reader
takes it as the first argument (the analog of the Leica `client`). It owns a background asyncio event
loop and the gRPC channel. Always `close()` it (or use it within your own `try/finally`).

**Blocking facade over an async core.** ZEN API is async; `zenapi` runs a persistent event-loop
thread inside `ZenClient` and bridges it: unary RPCs via `submit()`, server streams via `stream()`.
You never write `await`.

**Micrometers in, meters on the wire.** All public positions/tolerances are micrometers. Conversion
to SI meters happens only where a request is built or a reply parsed.

**Result dictionaries.** Commands return a dict with a stable envelope plus command-specific keys:

| Key | Meaning |
|---|---|
| `success` | `True` if the command was accepted (and, for acquisition, completed). Only a transport/RPC error makes this `False`. |
| `confirmed` | `True`/`False` if a readback confirmation ran; `None` if none was configured. |
| `message` | Human-readable summary. |
| `timing` | `{pre_check_s, fire_s, confirm_s, total_s, attempts, confirm_attempts, method}`. |
| `logs` | Ordered list of `{ts, level, msg}` entries. |
| *(command-specific)* | `position` (move_xy), `z_um` (move_z), `index` (set_objective), `output_name` (run_experiment). |

**Confirmation model.** `success=True, confirmed=False` means "the command was sent and did not
error, but the driver could not verify the resulting state." This is *not* a failure — it mirrors the
Leica posture. For acquisitions, the real data gate is `save()`, not the confirmation.

**Errors.** A failed RPC raises `grpclib.GRPCError`; the driver classifies the status code as
transient (retried) or permanent (fails fast). Limit violations and bad arguments raise before any RPC.

## 6. API reference

All functions are synchronous. `client` is a `ZenClient`.

### Connection

```python
connect(config_path=None, *, host=None, port=None, cert_file=None,
        control_token=None, connect_timeout=None) -> ZenClient
```
Connect through the gateway and return a ping-verified client. Explicit kwargs override `config.ini`;
all four connection parameters must resolve from somewhere. **Raises** `ValueError` (missing
parameter), `ConnectionError` (channel built but ping failed), `RuntimeError` (`zen_api` not installed).

```python
close(client) -> None          # close the channel and stop the event loop (idempotent)
ping(client) -> bool           # cheap connectivity check (attempts a stage read)
```

### Motion — stage & focus

```python
move_xy(client, x, y, unit="um", *, max_retries=None, tolerance=None) -> dict
```
Move the XY stage to an absolute position. `unit` ∈ `{"um","mm","m"}`. Enforces the configured stage
limits (in µm) **before** any RPC; out-of-range returns `success=False` without contacting ZEN. Result
carries `position` (`{x_m,y_m,x_um,y_um}`). `tolerance` overrides the confirm tolerance (µm).

```python
move_z(client, z, unit="um", *, max_retries=None, tolerance=None) -> dict
```
Move focus (Z) to an absolute position. Result carries `z_um`.

```python
move_xy_with_backlash(client, x_um, y_um, *, overshoot_um=50.0, settle_ms=100, tolerance=None) -> dict
correct_backlash(client, *, overshoot_um=50.0, settle_ms=100, tolerance_um=20.0) -> dict
```
Approach a target from a fixed −X/−Y direction to take up mechanical slack for repeatable positioning.

### Objective

```python
set_objective(client, *, index=None, name=None, magnification=None, max_retries=None) -> dict
```
Switch the objective. Provide exactly one selector: turret `index` (int), `name` (str), or
`magnification`. Names/magnifications are resolved against the fitted-objectives list. Result carries
the resolved `index`. **Raises** `ValueError` if the selector matches nothing.

### State readers

```python
get_xy(client, *, diagnostics=False) -> dict     # {"x_m","y_m","x_um","y_um"}
get_z(client, *, diagnostics=False) -> float      # micrometers
get_objective(client, *, diagnostics=False) -> dict   # {"index","name","magnification"}
get_objectives(client) -> list[dict]              # fitted objectives (cached on the client)
get_status(client, experiment=None, *, diagnostics=False) -> dict
ping(client) -> bool
```
With `diagnostics=True`, a reader returns a `Reading` (`.value`, `.source`, `.observed_at`) instead of
the bare value — used internally by confirmations for freshness gating. `get_status` without an
`experiment` returns a "not running" sentinel (a bare-instrument status stream is not exposed by the
API); with one, it snapshots the first status message.

### Acquisition

```python
load_experiment(client, name_or_path) -> Experiment
```
Load a ZEN experiment and return its handle (`Experiment(experiment_id, name)`).

```python
run_snap(client, experiment, *, poll_timeout=None, start_timeout=None) -> dict
run_experiment(client, experiment, *, output_name=None, poll_timeout=None,
               start_timeout=None, heartbeat_interval=None) -> dict
```
Acquire a single snap, or run a full experiment to completion (blocking via the status stream).
`run_experiment` result carries `output_name` (the CZI name ZEN wrote).

```python
acquire(client, experiment, *, mode="experiment", **kwargs) -> AcquisitionResult
```
High-level acquire: `mode="experiment"` (default) or `mode="snap"`. **Raises** `RuntimeError` if the
command did not succeed. Returns `AcquisitionResult(experiment_id, output_name, started_at,
finished_at, command_result)`.

```python
save(client, acq, output_root, naming, *, stable_timeout_s=60.0, stable_poll_s=0.5) -> SavedAcquisition
```
Resolve the CZI ZEN wrote (`get_image_output_path`), wait for the file to stop growing, and copy it
into `output_root/<acquisition_type>/data/` using the lab-wide
[`Naming`](../../../shared/output_layout/naming.py) slots (`.czi` extension). Returns
`SavedAcquisition(czi_path, naming)`.

### Monitoring

```python
monitor(client, experiment, *, kind="status", channel_index=0, enable_raw_data=False)
```
A blocking generator over a ZEN server stream. `kind="status"` yields status dicts until the stream
ends. `kind="pixels"` (the PixelStream → numpy frame path) raises `NotImplementedError` — it is an
[extension seam](#11-extension-seams).

### Stage limits

```python
set_stage_limits(*, x_min, x_max, y_min, y_max, z_min, z_max) -> None
get_stage_limits() -> dict
apply_stage_limits_from_config(stage_cfg: dict) -> None      # from load_stage_config(...)
load_stage_config(path) -> dict                              # validates {"stage_um": {x,y,z: [min,max]}}
```

## 7. Architecture

```
zmart_drivers/zeiss/zenapi/
├── connection/   zen_runtime.py  wheel check, TLS, metadata; the ONE place with zen_api paths/field names
│                 client.py       ZenClient: loop thread + submit()/stream() bridge + lazy stubs
│                 session.py      connect() / close()
├── commands/     dispatch.py     confirm_and_fire backbone (pre-check → fire_fn → confirm → re-fire + retry)
│                 errors.py       gRPC status-code classification
│                 confirmations.py readbacks + confirm_acquire (status-stream consumer)
│                 objectives.py   name/magnification → turret index
│                 commands.py     move_xy/move_z/set_objective/load_experiment/run_snap/run_experiment
├── readers/      api_reader.py   unary reads → dicts (the single m→µm boundary); monitor()
│                 reading.py      Reading envelope + freshness gate
├── config/       profiles.py     ZenApiProfile, ReaderProfile, CommandProfile + per-command instances
├── motion/       limits.py, movement.py, stage_config.py
├── acquisition/  product.py (neutral types), capture.py (acquire), save.py (CZI)
└── tests/        unit/ + helpers/mock_zen_api.py + hardware/ (marked, excluded by default)
```

**The one load-bearing adaptation.** The Leica dispatch fires by mutating a stateful model, calling a
synchronous transport, then reading an echo model for errors. ZEN has none of that: a command is
`await stub.<verb>(<Request>)`, which returns on success or raises `GRPCError`. So all of the
backbone's transport machinery collapses into a single vendor-supplied **`fire_fn`** — a zero-arg
synchronous callable (built per command in `commands.py`) that bridges one coroutine via
`client.submit(...)` and returns the standard result shape. `dispatch.py` stays completely
vendor-neutral; it never sees asyncio, gRPC, or stubs.

**Readers are api-only.** ZEN has no log to tail — errors are gRPC status codes and state comes from
unary reads plus the native status stream — so the Leica api/log/hybrid reader router collapses to a
single api path. (A real router can return if a hardware-event stream is added later.)

## 8. Configuration & tuning (profiles)

All per-command tuning lives in `config/profiles.py` as frozen `CommandProfile` instances; wrappers
accept explicit overrides only for tests/unusual runs.

| Profile | Posture |
|---|---|
| `STAGE_MOVE` | confirm within `1.0 µm`; single confirm window, no re-fire (ZEN `move_to` awaits completion). |
| `FOCUS_MOVE` | confirm within `0.5 µm`; single window, no re-fire. |
| `OBJECTIVE` | confirm index match; single window, no re-fire. |
| `SNAP` / `RUN_EXPERIMENT` | `max_retries=0`, never re-fire (a re-fire would start a second acquisition); unconfirmed ≠ failure; `call_timeout=None` (the status stream is the completion gate). |

`ZEN_API` (connection timing) and `READERS` (per-read deadlines) are the other tuning surfaces. The
`CommandProfile.__post_init__` guard forbids the incoherent `max_confirm_attempts==1` +
`refire_on_unconfirmed=True` combination.

## 9. Testing

```bash
python run_ci.py                # ruff (if present) + offline pytest + coverage (if present)
python run_ci.py --hardware     # ONLY the @pytest.mark.hardware suite (needs a live ZEN)
pytest -m "not hardware"        # offline suite directly
```

The offline suite (**50 tests**) runs with **no `zen_api` wheel, no gateway, no scope**: a behavioral
fake ZEN API (`tests/helpers/mock_zen_api.py`) is injected into a **real** `ZenClient`, so the
async→blocking bridge, dispatch retry/confirm loop, µm↔m conversion, limit enforcement, gRPC-error
classification, status-stream consumption, `acquire()`/`save()`, and clean package import are all
exercised for real — only the wire is faked. Hardware tests are marked and excluded by default; point
`ZENAPI_CONFIG` at a `config.ini` and run `pytest -m hardware` on a ZEN bench.

## 10. Risks / bench-verify

Transcribed from the OAD examples, not yet confirmed against the wheel or hardware. Each is confined so
a fix touches one place:

1. **Request/response field names & module paths** (`zen_api.lm.hardware.v2` vs `hardware.v1`, enum
   spellings, `experiment_id`) → `connection/zen_runtime.py`, `readers/api_reader.py`.
2. **Completion semantics** — the design assumes `move_to` awaits to motion-complete. If it returns on
   command-*accepted*, make the readback confirm mandatory (raise `confirm_poll_s`, set
   `refire_on_unconfirmed=True`) in `config/profiles.py`.
3. **CZI retrieval** — `get_image_output_path` signature; local-vs-share path from the driver host;
   whether `run_experiment` returns only after the CZI is fully written → `acquisition/save.py`.
4. **Status stream** — it takes an `experiment_id`; whether a bare-instrument idle stream exists and
   whether the server replays current status on subscribe (the "register before fire" ordering in
   `confirm_acquire`).
5. **TLS/auth & config schema** — validate against `zen_api_utils.misc.initialize_zenapi`.

## 11. Extension seams

Not built in the MVP, with clear insertion points:

- **Experiment/template editing** (analog of the Leica scanfields).
- **Calibration** — image↔stage affine, objective parfocality.
- **Pixel-pull exporter** — `monitor(kind="pixels")` → `ExperimentStreamingService` → numpy → OME/TIFF,
  plus a pluggable save exporter.
- **Autofocus** — software autofocus (`ExperimentSwAutofocusService`) and Definite Focus.
- **More devices** — Optovar (identical shape to the objective changer), filters, LEDs, detectors.
