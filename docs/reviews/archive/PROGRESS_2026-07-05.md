## 2. Maintainer decisions recorded (`MAINTAINER_DECISIONS.md`)

1. Hybrid reader is essential; api/log stay standalone; **hybrid is now the default**
   for routed reads. Rationale: api/log staleness is complementary; for changes the
   target is known, so one admissible witness suffices.
2. Backlash = simple go-and-return **procedure outside acquisition** in the driver;
   exposed as a Leica acquisition option in the controller. (Not yet implemented.)
3. Test strategy: three tiers (hardware / mock hardware / offline); adapter must be
   tested **through the controller**.
4. Orphan test scripts may be deleted (done).
5. `experimental/` may be reworked without bench validation.
6. `confirmed` is best-effort — 3 retries then report unconfirmed and continue —
   **except acquire's idle gate, which must be confirmed** (and still needs a real
   deadline so a dead LAS X errors instead of hanging).
7. **Limits enforced at the lowest layer** (command wrappers, so nothing on top can
   bypass); **no bundled default limits for enforcement** (templates only; the
   `set_stage_limits` notebook is the file factory; connect verifies placement);
   hardcoded physical backstop for the motoric stage; explicit `null` = deliberately
   unlimited, absent key fails closed; calibration co-located but applied at the
   objective-change path. Implemented (`474c3a6`, `b90b92f`); plan +
   two review docs under `docs/design/`.

## 7. Post-bench fixes, round 2 (2026-07-06)

Three more §6 items closed, plus one closed by maintainer decision instead of code.

**Backlash wired into the acquisition option, tested at all three tiers.** It turned
out `zmart_adapter.acquire()` already called `_motion.correct_backlash()` when
`resolved["backlash_correction"]` is true, using the baked-in-default motion
primitives — decision §2b (bench day) had already retired the "thread calibrated
backlash values" question, since there is no calibrated backlash to thread. The real
gap was coverage: only the `False` path was tested. Added an adapter-direct unit test
(order: select → backlash → capture; `settle == "backlash-corrected"`), a
through-the-controller offline test (decision §3: opens a real
`zmart_controller.Session` against the in-process mock CAM, patching only the
`_capture`/`_save` I/O boundary a mock can't satisfy), and extended the real-hardware
validator's acquire phase to assert it live. Verified live on the real STELLARIS
twice (different exporters): `backlash_correction=True` threaded correctly through
the whole pre-save sequence both times; both runs then hit an unrelated,
pre-existing export-file-detection failure (native autosave and `navigator_expert`
exporters both failed to find their output files on this machine) — a separate
environmental issue, not touched here.

**CAM teardown on disconnect — fixed, and the "unverified" half of CF-06 resolved.**
`commands/gate.py` already had `uninstall(client)` to drop a client's `FunctionLimits`
from the module-level gate registry, but nothing called it — `disconnect()` was
flag-only. Wired it in. Then went further on the two things CF-06 flagged as
*unverified* rather than assumed:
- **No CAM-level teardown exists to call, confirmed by reflection** — the client
  `connect_python_client` returns (`LasxApiClientPyModel`) exposes only
  `Connect`/`ConnectAsync` (plus a COM-style `Release`, not a session close). There is
  no `Disconnect`/`Close`/`Dispose` for our code to invoke, even in principle.
- **Double-connect is safe, verified live** — connecting twice under the same client
  name without disconnecting the first was untested behavior per CF-06. Tested it
  directly against the real instrument: both connections independently ping and read
  the correct selected job, and the first stays fully usable after the second
  connects. It is a resource leak (nothing releases the old `.NET`-side registration)
  but not a corruption or dead-end risk.

Both findings are now recorded in the `disconnect()` docstring. Added an offline unit
test (`gate.state_for` is installed after connect, `None` after disconnect) and
confirmed the same live.

**`require_canonical_scan_orientation` removed, not wired in.** The obvious next step
per §6 was to wire this into `connect()` as a fail-closed gate. Before doing that, it
was checked against the real instrument's actual settings — and it raised:
ZMB-LASX-PC's live `ImageTransformation` is `RIGHTTOP`, not `TOPLEFT`, on the
**currently installed LAS X version**. Wiring the check in as fail-closed would have
immediately blocked every `connect()` on this machine. Maintainer call: remove the
check entirely rather than wire it in or leave it dormant — **the next LAS X version
does not have this transform**, so this is not tracked as an open decision; no
settings change or math change is expected once that version lands. Swept all
references (`navigator_expert/__init__.py` re-export, README quick-start/API-table/
gotchas list, the definition in `connection/session.py`). `get_lasx_settings()`
itself is untouched — still used by `navigator_expert_export.py` and
`experimental/lrp_edits/roi.py`.

**Acquire idle-gate: closed by decision, not code.** The recommended fix (a real
deadline on `check_idle`/`confirm_acquire`'s idle wait, per CF-02/CF-03) was rejected.
Maintainer decision, recorded in `MAINTAINER_DECISIONS.md` §6: the idle wait stays
**unbounded, deliberately, permanently** — a dead LAS X hanging forever is an
acceptable failure mode (nothing recoverable can be done about a dead LAS X anyway);
a deadline that aborts a slow-but-alive real acquisition is not acceptable, and a real
acquisition can legitimately take an arbitrarily long time. This is a narrow exception
to command confirmation's general 3-retry/best-effort rule (decision §6, first
bullet) — it does not change any other wait in the codebase.

Verified throughout: offline suite 866 passed, 0 failed, ruff clean after every change
in this round (not a regression check against §3's 906 — that run used a different
`-k` filter and env, not a comparable count).
